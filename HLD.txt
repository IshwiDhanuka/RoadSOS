1. Problem Statement
1.1 Context
India records over 4.5 lakh road accidents and 1.5 lakh fatalities every year, making it one of the highest road-accident mortality rates in the world. A large proportion of these deaths are preventable — they occur not because medical care is unavailable, but because it is not reached within the golden hour, the 60-minute window immediately after a trauma event during which timely intervention can save lives and significantly reduce injury severity.
When an accident occurs, the people present — victims, bystanders, and passing motorists — face a critical information problem. They do not know:
Which hospital or trauma centre is nearest and equipped for accident injuries
How to reach an ambulance service specific to that district or highway corridor
Where the nearest police station is to file an FIR or get emergency assistance
How to contact towing services or vehicle rescue providers at that location
Any of the above when they have no internet connectivity — tunnels, rural highways, dead zones
This information gap causes critical delays. Every minute spent searching for a number or figuring out the nearest hospital is a minute not spent getting the victim to care.

2. Proposed Solution
2.1 Product Summary
ROADSoS is a single-tap emergency response Android application. A person at the scene of a road accident opens the app, taps one large SOS button, and within 2 seconds sees a prioritised list of the nearest trauma centres, ambulances, police stations, towing services, and puncture shops — with one-tap call and navigation to each.
Beyond service discovery, ROADSoS solves the offline problem with a two-layer approach: a pre-downloaded SQLite cache covering the user's area, and a Bluetooth mesh relay that forwards a signed SOS packet through nearby phones to the internet when the victim's own device has no connectivity. A built-in first-aid guide works with zero network dependency.

3. System Architecture
3.1 Three-Component Overview
Component A  —  Android App (React Native)
6 screens: SOS, Map, Search, First Aid, Report, Offline
Redux store for online state; SQLite for offline cache
Calls backend via axios / react-query
Calls MANET layer via VANETBridge.ts → Kotlin Native Module
Component B  —  Express Backend (Node.js on Render.com)
Single Express app — 7 routes, no microservices overhead
Upstash Redis — geohash response cache, sub-millisecond hits
Firestore — verified seed DB + SOS events + device keys + nonces + users
Google Places API — live service discovery, global
FCM — push notifications to nearby users, free and unlimited
Component C  —  MANET Mesh Layer (Kotlin, on-device)
BLE GATT server/client — primary device-to-device SOS relay
Wi-Fi Direct — secondary relay, ~200 m range fallback
ECDSA-P256 packet signing via Android KeyStore
AES-256-GCM GPS encryption — relay nodes never see plaintext location
Hop-chain signatures — full audit trail of relay path
3.2 Why MANET Instead of VANET
The problem statement is set in India. VANET (Vehicle-to-Vehicle communication via IEEE 802.11p) requires dedicated V2X radios installed in vehicles — hardware that does not exist in any Indian passenger vehicle today and has no deployment timeline. MANET (Mobile Ad-hoc Network) using Bluetooth and Wi-Fi Direct achieves identical security and mesh-relay properties using hardware already present in every Android phone. The MANET implementation directly maps to all VanSec elective assessment criteria and can be upgraded to VANET in future by swapping only the transport layer.

4.2 SOSPacket (MANET)
// Kotlin data class — serialized to JSON via GSON for GATT transfer
data class SOSPacket(
  val eventId:     String,   // UUID v4
  val userId:      String,   // SHA-256 of Android device ID
  val locationEnc: String,   // AES-256-GCM encrypted 'lat,lng', base64
  val encKey:      String,   // AES key wrapped with server RSA-2048-OAEP, base64
  val iv:          String,   // AES-GCM IV, base64
  val timestamp:   Long,     // epoch ms
  val nonce:       String,   // 128-bit SecureRandom hex — replay prevention
  val signature:   String,   // ECDSA-P256 over 'eventId|timestamp|nonce|locationEnc'
  val hopCount:    Int,      // 0 = origin device, max 5
  val relayChain:  List<RelayHop>
)
data class RelayHop(
  val nodeId:  String,  // SHA-256 of relay device ID
  val hopSig:  String,  // ECDSA over 'eventId|hopIndex|ts'
  val ts:      Long
)

6. MANET Security Architecture
6.1 Threat Model
Threat
Attack Scenario
Severity
Fake SOS injection
Attacker broadcasts forged SOSPacket to waste emergency responder time and resources
Critical
Replay attack
Attacker captures a valid packet and re-broadcasts it to trigger duplicate alerts
High
Location stalking
Relay node reads victim GPS from packet to track location
High
Packet tampering
Attacker modifies hopCount, timestamp, or location in transit
High
Hop flooding
Malicious relay increments hopCount to 0, making packet loop indefinitely
Medium
Sybil attack
Attacker creates multiple fake relay identities to poison hop-chain audit trail
Medium
6.2 Countermeasures
Threat
Countermeasure
Implementation File
Fake SOS
ECDSA-P256 signature on every packet. Server fetches sender public key from Firestore and verifies before creating any SOSEvent. 403 on failure.
SOSPacketSigner.kt, ECDSAVerifier.ts
Replay attack
128-bit cryptographically random nonce included in signed payload. Server stores nonce in Firestore with 60 s TTL. 409 on duplicate. Relay nodes maintain LRU cache of 500 seen nonces.
NonceGenerator.kt, NonceCache.kt
Location stalking
GPS encrypted with AES-256-GCM using a per-event ephemeral key. AES key itself encrypted with server RSA-2048-OAEP public key (pinned in APK). Relay nodes only ever see base64 ciphertext.
LocationEncryptor.kt
Packet tampering
ECDSA signature covers eventId|timestamp|nonce|locationEnc. Any modification invalidates signature.
SOSPacketSigner.kt
Hop flooding
hopCount field incremented by each relay. RelayManager drops packet without forwarding if hopCount ≥ 5.
RelayManager.kt
Sybil attack
Each relay appends a RelayHop signed with its own Android KeyStore key. Server verifies every hop signature against registered public keys. Unregistered relay flagged.
HopChainManager.kt, HopChainVerifier.ts
6.3 Key Management
Device key pair: EC P-256 generated at first app launch using Android KeyStore API (TEE-backed on modern devices). Never leaves the device.
Public key registration: uploaded to Firestore /devices/{deviceId}/publicKey on first launch. Used by server and relay nodes to verify signatures.
Server RSA key pair: RSA-2048 key pair generated once. Private key stored as environment variable on Render. Public key embedded in APK at build time (certificate pinning).
AES key per SOS: ephemeral AES-256 key generated fresh for every SOS event. Encrypted with server RSA public key. Only server can decrypt to read victim GPS.
